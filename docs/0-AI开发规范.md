# 前端项目AI开发操作规范

## 一、目的

为规范前端项目中AI工具的使用流程，平衡开发效率与代码质量，明确AI使用边界，保障项目安全与可维护性，特制定本规范。



## 二、适用范围

本规范适用于所有使用AI工具（如ChatGPT、Cursor、GitHub Copilot等）进行前端开发的人员及相关项目。


## 三、AI开发使用边界

### 3.1 推荐使用AI开发的场景

1. **代码生成**：
   - 基于明确的需求描述，生成单一功能的前端代码片段（如组件逻辑、接口请求封装、简单的交互逻辑等）；
   - ❗️ 仅生成最小可执行单元代码，禁止生成完整页面 / 模块代码。
2. **代码优化**：
   - 对已有的前端代码进行性能优化、语法规范化、冗余代码清理等（需人工验证优化效果及原有功能）；
   - ❗️ 优化后必须人工验证功能无异常，禁止直接替换原有核心代码。
3. **文档编写**：
   - 生成组件使用文档、技术方案说明文档、开发注释模板等；
   - ❗️ 不得包含敏感信息。
4. **注释编写**：
   - 为已完成的代码补充规范化注释（如函数注释、组件注释、复杂逻辑注释）。
5. **测试用例生成**：
   - 针对单一功能/组件生成单元测试、端到端测试用例。
6. **Bug修复**：
   - 针对明确的前端Bug（如语法错误、逻辑异常、兼容性问题、运行错误），生成修复方案或代码片段；
   - ❗️ 修复前需复现 Bug，修复后验证无新问题引入。
7. **跨端适配代码转换**：
   - 将已有的 Web 端代码转换为小程序、移动端 H5 适配代码，减少重复开发工作量；
   - ❗️ 转换后需验证跨端兼容性，禁止直接上线未测试的转换代码。
8. **数据可视化图表配置**：
   - 基于指定数据源和图表类型（如柱状图、折线图、热力图），生成 ECharts/AntV 等可视化库的配置代码，快速实现图表渲染；
   - ❗️ 
9. **正则表达式编写与调试**：
   - 针对复杂的字符串匹配、替换需求（如表单验证、数据格式化），生成正则表达式并提供调试用例；
10. **API 请求封装与 Mock 数据生成**：

   - 根据后端 API 文档，生成前端请求函数封装代码，同时生成符合接口格式的 Mock 数据，用于前端联调前的功能自测。

11. **相似功能开发**：

   - 提供已实现的 A 页面代码 / 功能逻辑作为参考，告知 AI B 页面的差异化需求，生成符合项目规范的 B 页面代码（需人工核对页面结构、样式及逻辑一致性）；
   - ❗️ 需明确告知差异化需求，生成后必须核对结构、样式、逻辑与项目规范的一致性。

### 3.2 不允许使用AI开发的场景

1. **架构设计**：禁止依赖AI完成前端项目整体架构、模块划分、状态管理方案等核心架构设计；
2. **代码审查**：禁止将AI的判断作为代码审查的最终结论，代码审查必须由研发人员人工完成；
3. **敏感信息处理**：禁止向AI工具输入项目敏感信息（如接口密钥、数据库账号、用户隐私数据、业务核心逻辑、公司内部规范等）；
4. 补充说明：若AI生成内容涉及敏感信息相关逻辑，需人工完全重写，且不得将敏感信息传入AI训练/生成环节。

## 四、AI开发操作流程规范

### 4.1 需求拆解要求

1. 禁止直接使用 AI 生成完整大需求的代码，必须先将需求拆解为最小可执行单元（如单个组件、单个交互逻辑、单个接口对接逻辑等）；
2. 研发人员需先自主搭建页面整体框架（如路由结构、组件层级、基础布局、状态管理骨架），再针对拆解后的小需求使用 AI 实现具体功能；
3. 需求拆解需形成书面 / 线上记录（如需求拆解清单），明确每个小需求的功能边界、输入输出要求、依赖关系，作为 AI 使用的依据；
4. 需求拆解完成后，需仔细审核确认，确保拆解粒度合理、无遗漏关键逻辑。


### 4.2 UI实现细节要求

1. 使用AI生成UI相关代码（如样式、组件布局）后，必须人工核对所有视觉细节，包括但不限于：
   - 阴影的大小、颜色、透明度；
   - 圆角的像素值；
   - 间距（margin/padding）的精准度；
   - 字体、字号、行高、颜色；
   - 适配不同屏幕尺寸的展示效果；
2. 视觉细节需与UI设计稿100%匹配，AI生成的样式偏差需人工修正，禁止直接使用未核对的AI生成样式代码。

### 4.3 AI生成代码验证要求

1. **代码结构检查**：AI生成代码后，首先人工检查代码结构，包括：
   - 代码是否符合项目编码规范（如命名规则、代码缩进、文件结构）；
   - 是否存在冗余代码、死代码、未定义变量；
   - 依赖引入是否合理，是否存在未使用的依赖；
   - 逻辑是否符合前端开发最佳实践（如避免内存泄漏、合理使用异步逻辑）；
2. **功能验证**：代码结构检查通过后，需在开发环境中运行代码，验证功能是否符合需求，包括：
   - 交互逻辑是否正常触发；
   - 数据渲染是否准确；
   - 边界条件（如空数据、异常输入）是否处理；
   - 兼容性是否满足项目要求（如浏览器版本、移动端机型）；
3. **安全验证**：针对涉及数据交互、权限相关的 AI 生成代码，需额外验证：
   - 是否存在 XSS、CSRF 等安全漏洞；
   - 数据传输是否符合项目加密规范；
   - 权限校验逻辑是否完整；
4. **代码保存与回退**：验证过程中需及时保存每一个可用版本的代码（如通过Git提交、本地备份），明确版本标识，便于出现问题时快速回退到上一可用版本。
   - 禁止跳过验证环节直接提交 AI 生成的代码。

### 4.4 代码接纳（Accept）要求

1. 尽量逐行/逐功能块接纳AI生成的代码，禁止一次性接纳多个文件的全部变更；

2. 接纳代码前需明确每一行代码的作用，对不理解的代码片段需先验证、注释，再接纳；

3. 接纳后的代码需整合到项目现有代码体系中，确保与原有代码无冲突，且符合项目整体规范。

   

### 总结

1. AI使用需明确边界：仅用于代码生成、优化等6类场景，禁止用于架构设计、敏感信息处理等3类场景；
2. 核心操作要求：先拆解需求搭框架，再用AI实现小模块，UI细节必须人工核对，代码需逐行验收并做好版本保存；
3. 责任归属：研发人员对AI生成代码的质量、安全、合规性负最终责任，人工审核是不可省略的核心环节。
